    Getting started with git:

    Install Git. Then run the following commands in CLI to set configurations.

    • Setting up local system
      •             git config --global user.email "your@email.id"
            git config --global user.name "your name"
    • Set your prefered editor
      •             git config --global core.editor vim
    • Verify the changes made
      •             git config --global --list

    Getting started with github:

    Repositories or the repos can be created in local system CLI and also in github.The simplest method is to create repo in github and then clone it into local system. This can be done as follows

    • Create an access token in developer settings < classic tokens
    • Create a new repo
    • Copy the web URL of the newly created repo
    • In the local system CLI, use the command
      •             git clone https://accesstoken@remainingPartOfWebURL
    • Once cloning is done, to update changes use
      •             git pull
    • After making changes to repo, use
      •             git status
            git add .
            git status
            git commit -m "any commit message you wish"
            git push -u origin main

    The Flow:

    Initiating the project:

    To work with git and github, you need some files for which the version control can be applied. Lets consider different possibilities of initiating the project.

    1. Starting from scratch:
      2.             git init project-name
    2. Existing project:
      3.             #Get in to the main directory of the project. Use command
            git init
    3. Forking and cloning: Once the cloning is done, the flow is same as mentioned for "Getting started with github".

    Tracking files:

    • Files in and not in staging area can be checked by
      •             git status
    • To add files into staging area
      •             git add filename
            
            #To add all the changed files
            git add .
    • To list the files that git is tracking
      •             git ls-files
    • Files, once in the staging area can be commited using
      •             git commit -m "Any apt message"
            # To amend the commit
            git commit --amend
    • To commit the files already being tracked, with out adding to staged area
      •             git commit -am "any apt message"
            Note: This doesnt include newly created files, not staged atleast once.

    Backing out files:

    • You have made the changes, moved to staging area and then want to unstage.
      •             git reset HEAD filename
    • To completely discard changes in a unstaged file.
      •             git checkout -- filename

    Renaming and moving files:

    When a file is renamed, at OS level, git considers it as a file delted and new file added. And it wont be in staging area. This adds another step of moving of files in to staging area using git add filename. To avoid this, we can rename files using git.

                git mv oldname.txt newname.txt

    Deleting files

    Similar to renaming, files can also be deleted using git and save an extra step using

                git rm filename
            # -r and -f options apply.
            Note: Changes made before commit wont be saved.

    History:

    git log
    command is used to display the commit history. Some important options are:

    • git log --abbrev-commit
       - Displays all commits with a shorter commit ID 
    • git log --oneline --graph --decorate
       - More human readable form. 
    • git log abbreCommit1...abbreCommit2
       - Displays commits between the abbreviated commit IDs provided (range). 
    • git log --since="3 days ago"
       - Displays commits from the last 3 days. 3 can be any number. 
    • git log -- filename
       - Displays commits specific to file with name filename. 
    • git log --follow -- filename
       - Follows renames / changes for a specific file with name filename. 
    • git show fullcommitID
       - Shows complete information on the particular commit.

    The .gitignore file:

    Any file that is not required to be be tracked can be added to .gitignore file

    Branching:

    • To list all the available branches
      •             git branch
            git branch -l
            git branch -a
    • Creating a new branch
      •             git branch branchname
    • Checking out to a branch
      •             git checkout branchname
            #To create a new branch and checkout to it
            git checkout -b newbranch
    • Renaming a branch
      •             git branch -m oldname newname
    • To delete a branch, you have to be on a different branch first. Then the command
      •             git branch -d branchname
    • Merging a branch with main
      •             git checkout main
            git merge branchname
            #To preserve branch line
            git merge branchname --no-ff
            #When merge results in a commit
            git merge branchname -m "apt message"

    Rebasing:

    Rebasing a branch is basically syncing the branch with main and getting all the changes on main to the current branch. For that, git will first rewind all the changes on current branch, apply the changes from main branch on to current branch and then reapply the changes made on the current branch.

                git rebase main 
            #To abort a rebase
            git rebase --abort

    If the local repo has to be updated with the remote repo changes just like rebasing

                git fetch origin main
            git pull --rebase origin main

    Stashings:

    Stashing in git is used to save a work inprogress and make the working tree clean temporarily so as to work on a different file. The stashed file will not be displayed in git status and lets you work on a different file. Once, the user finishes working on other files, he can bring in the stashed file and work on it.

                git stash
            #For multiple stashes
            git stash save "stash name"
            # To stash untracked files
            git stash -u
            git stash apply
            git stash list
            git stash drop
            # Apply and drop can be done in one command.
            git stash pop
            # When there are multiple stashes - n stashes
            git stash list
            git stash show stash@{0} #0-n number can be used.
            git stash apply stash@{0} #0-n number can be used
            git stash drop stash@{0} #0-n number can be used
            git stash clear
            # To move the stash to a different branch
            git stash branch newbranchname

    Tagging is just like a label that can be added to commit so as to represent some milestones etc. Tag are 2 types:

    1. Light weight tags:
      2.             git tag tagname
            git tag tagname -m "tag mesage"
            git tag --list
            git show tagname
            git tag --delete tagname
    2. Annotated tags:
      3.             git tag -a tagname
            # Differece between tags
            git diff tag1 tag2
            # To tag a previous commit
            git tag -a tagname commitidabbrev
            # To change the commit associated with a tag
            git tag -a tagname -f correctcommitidabbrev
            # To push tag to repository
            git push origin tagname
            # Push all tags to the repository
            git push origin main --tags
            # To not push a tag to a repository
            git push origin :tagname

    Reset and Reflog:

    • To reset the previous commits
      •             git reset HEAD^
            # To reset multiple commits
            git reset HEAD^n            # n is the number of commits to reset
                
    • To list all the commits and resets
      •  
                  git reflog 
      
                
    • To go back on resets
      • 
                
                  git reset reflogIDabbrev
            # Or
            git reset head@{n}          # n is from reflog
